Node Affinity vs Pod Anti-Affinity

Understanding Kubernetes Pod Scheduling Constraints

🔒 Required (Hard Constraint)

requiredDuringSchedulingIgnoredDuringExecution - Pod MUST satisfy the rule or it won't be scheduled. Used when placement is critical.

💡 Preferred (Soft Constraint)

preferredDuringSchedulingIgnoredDuringExecution - Scheduler tries to satisfy the rule but can ignore it if needed. Used for optimization.

🎯 Node Affinity

Attracts pods to specific nodes based on node labels. Controls WHERE a pod runs based on node characteristics.

Visual: Pod Attracted to Production Node

Example YAML:
apiVersion: v1
kind: Pod
metadata:
  name: production-pod
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: env
            operator: In
            values:
            - production
  containers:
  - name: nginx
    image: nginx

↔️ Pod Anti-Affinity

Spreads pods apart based on pod labels. Prevents pods from running on the same node/topology to ensure high availability.

Visual: Pods Spread Across Nodes

Worker Node 2

Example YAML:
apiVersion: v1
kind: Pod
metadata:
  name: staging-pod-2
spec:
  affinity:
    podAntiAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchExpressions:
          - key: env
            operator: In
            values:
            - staging
        topologyKey: kubernetes.io/hostname
  containers:
  - name: nginx
    image: nginx
Feature Node Affinity Pod Anti-Affinity
Purpose Attract pods to specific nodes Spread pods away from each other
Based On Node labels (node characteristics) Pod labels (pod relationships)
Use Case Hardware requirements (GPU, SSD), environment isolation (prod/dev) High availability, fault tolerance, spreading replicas
Operators In, NotIn, Exists, DoesNotExist, Gt, Lt In, NotIn, Exists, DoesNotExist
Topology Key Not used Required (e.g., kubernetes.io/hostname, topology.kubernetes.io/zone)
