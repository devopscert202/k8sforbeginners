# Upgrade guide: v1.30.x → v1.31.14

For upgrade prerequisites, version skew policy, etcd backup, drain/cordon order, validation, and rollback concepts, see **[Upgrade procedure](./upgrade_procedure.md)**.

This document lists **v1.31.14**-specific steps (package **`1.31.14-1.1`**) for a **kubeadm** cluster on **Debian/Ubuntu**-style hosts. Replace `worker-node-1.example.com`, `worker-node-2.example.com`, and similar with your real node names.

**Note:** The control-plane section below upgrades **`kubeadm`** and runs **`kubeadm upgrade apply`** before bumping **`kubelet`/`kubectl`** on the master. If you drain the control plane for kubelet upgrades, follow the drain/uncordon sequence in the [upgrade procedure](./upgrade_procedure.md).

---

## 1. Master node (control plane) upgrade

### 1.1 Setup v1.31 APT repository

```bash
curl -fsSL https://pkgs.k8s.io/core:/stable:/v1.31/deb/Release.key \
  | sudo gpg --dearmor -o /etc/apt/keyrings/kubernetes-apt-keyring.gpg

echo 'deb [signed-by=/etc/apt/keyrings/kubernetes-apt-keyring.gpg] \
https://pkgs.k8s.io/core:/stable:/v1.31/deb/ /' \
  | sudo tee /etc/apt/sources.list.d/kubernetes.list

sudo apt-get update
apt-cache madison kubeadm
```

### 1.2 Install kubeadm 1.31.14

```bash
sudo apt-mark unhold kubeadm
sudo apt-get install -y kubeadm=1.31.14-1.1
sudo apt-mark hold kubeadm

kubeadm version
```

### 1.3 Upgrade control plane

```bash
sudo kubeadm upgrade apply v1.31.14
```

If you hit timeout or known preflight issues, you may use (review logs carefully):

```bash
sudo kubeadm upgrade apply v1.31.14 --ignore-preflight-errors=all
```

### 1.4 Upgrade kubelet and kubectl (master)

```bash
sudo apt-mark unhold kubelet kubectl
sudo apt-get install -y kubelet=1.31.14-1.1 kubectl=1.31.14-1.1
sudo apt-mark hold kubelet kubectl

sudo systemctl daemon-reload
sudo systemctl restart kubelet
```

## 2. Worker node 1 upgrade

### 2.1 Setup v1.31 APT repository (on worker-1)

```bash

  | sudo gpg --dearmor -o /etc/apt/keyrings/kubernetes-apt-keyring.gpg

  | sudo tee /etc/apt/sources.list.d/kubernetes.list

sudo apt-get update
apt-cache madison kubeadm
```

### 2.2 Install kubeadm 1.31.14 (on worker-1)

```bash
sudo apt-mark unhold kubeadm
sudo apt-get install -y kubeadm=1.31.14-1.1
sudo apt-mark hold kubeadm
```

### 2.3 Upgrade node configuration (on worker-1)

```bash
sudo kubeadm upgrade node
```

### 2.4 Drain worker node (from admin / control plane)

```bash
kubectl drain worker-node-1.example.com \
  --ignore-daemonsets --delete-emptydir-data --force
```

### 2.5 Install kubelet and kubectl (on worker-1)

```bash
sudo apt-mark unhold kubelet kubectl
sudo apt-get install -y kubelet=1.31.14-1.1 kubectl=1.31.14-1.1
sudo apt-mark hold kubelet kubectl

sudo systemctl daemon-reload
sudo systemctl restart kubelet
```

### 2.6 Uncordon worker node (from admin / control plane)

```bash
kubectl uncordon worker-node-1.example.com
```

## 4. Post-upgrade verification

```bash
kubectl get nodes -o wide
kubectl get pods -A
kubectl version --short
kubeadm version
```

Confirm:

* All nodes report version **v1.31.14**
* Nodes show `Ready`
* System pods in `kube-system` are running as expected

---

## 5. v1.31-specific notes

* Use **`apt-mark hold`** on **`kubeadm` / `kubelet` / `kubectl`** at the target version if you want to prevent accidental upgrades.
* **HA control plane:** Upgrade each control plane member in sequence; use **`kubeadm upgrade node`** on members after the first (see [Upgrade procedure](./upgrade_procedure.md)).
* If **`kubeadm upgrade apply`** fails on preflight or timeouts, **`--ignore-preflight-errors=all`** is a last resort—always inspect logs.
