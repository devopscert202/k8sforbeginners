# Kubernetes Upgrade Guide: v1.31 → v1.32

## Overview

This guide walks you through upgrading a Kubernetes cluster from version **1.31** to **1.32**. The upgrade process should be performed carefully, following the recommended sequence: control plane first, then worker nodes.

---

## Prerequisites

- ✅ Current cluster running Kubernetes **v1.31**
- ✅ Backup of etcd database
- ✅ Tested upgrade in non-production environment
- ✅ Read [Kubernetes v1.32 Release Notes](https://github.com/kubernetes/kubernetes/blob/master/CHANGELOG/CHANGELOG-1.32.md)
- ✅ Verify all workloads are compatible with v1.32
- ✅ Check for deprecated API usage

---

## What's New in v1.32

### Major Features

1. **Gateway API v1.2 GA** - Advanced traffic management capabilities
2. **Pod Scheduling Improvements** - Enhanced scheduler performance
3. **Image Pull Policy Enhancements** - More granular control over image pulling
4. **Kubeadm Improvements** - Simplified cluster upgrades
5. **Security Enhancements** - Improved authentication and authorization

### API Changes

#### Graduated to Stable (GA)
- `gateway.networking.k8s.io/v1` - Gateway API core resources
- `ClusterTrustBundle` API - Certificate trust bundle management

#### Deprecations
- None in this release that affect core resources

#### Removed APIs
- No major API removals in v1.32

### Important Changes

1. **etcd Version**: Recommended etcd version is v3.5.15+
2. **Container Runtime**: containerd 1.7.x or CRI-O 1.31.x recommended
3. **Feature Gates**: Several beta features promoted to GA

## Pre-Upgrade Checklist

```bash
# 1. Check current cluster version
kubectl version --short

# 2. Check node status
kubectl get nodes -o wide

# 3. Check pod status across all namespaces
kubectl get pods -A

# 4. Check for deprecated API usage
kubectl get --raw /metrics | grep apiserver_requested_deprecated_apis

# 5. Backup etcd (critical!)
# See: https://kubernetes.io/docs/tasks/administer-cluster/configure-upgrade-etcd/#backing-up-an-etcd-cluster

# 6. Check component health
kubectl get componentstatuses

# 7. Drain and cordon nodes before upgrade (do this per node during upgrade)
# kubectl drain <node-name> --ignore-daemonsets --delete-emptydir-data
```

## Backup etcd

**⚠️ CRITICAL**: Always backup etcd before upgrading!

```bash
# Set etcd endpoint and certificates
ETCDCTL_API=3 etcdctl \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  snapshot save /backup/etcd-snapshot-$(date +%Y%m%d-%H%M%S).db

# Verify backup
ETCDCTL_API=3 etcdctl --write-out=table snapshot status /backup/etcd-snapshot-*.db
```

## Upgrade Process

### Step 1: Upgrade Control Plane

#### Using kubeadm

```bash
# On the first control plane node

# 1. Update package repositories
sudo apt update  # Debian/Ubuntu
# OR
sudo yum update  # RHEL/CentOS

# 2. Check available kubeadm version
sudo apt-cache madison kubeadm | grep 1.32
# OR
sudo yum list --showduplicates kubeadm --disableexcludes=kubernetes | grep 1.32

# 3. Upgrade kubeadm to v1.32.x
sudo apt-mark unhold kubeadm && \
sudo apt-get update && \
sudo apt-get install -y kubeadm=1.32.0-00 && \
sudo apt-mark hold kubeadm

# OR for RHEL/CentOS:
sudo yum install -y kubeadm-1.32.0-0 --disableexcludes=kubernetes

# 4. Verify kubeadm version
kubeadm version

# 5. Drain the control plane node
kubectl drain <control-plane-node> --ignore-daemonsets

# 6. Plan the upgrade
sudo kubeadm upgrade plan

# 7. Apply the upgrade
sudo kubeadm upgrade apply v1.32.0

# You should see output like:
# [upgrade/successful] SUCCESS! Your cluster was upgraded to "v1.32.0". Enjoy!

# 8. Uncordon the node
kubectl uncordon <control-plane-node>
```

#### Upgrade kubelet and kubectl on Control Plane

```bash
# 1. Upgrade kubelet and kubectl
sudo apt-mark unhold kubelet kubectl && \
sudo apt-get update && \
sudo apt-get install -y kubelet=1.32.0-00 kubectl=1.32.0-00 && \
sudo apt-mark hold kubelet kubectl

# OR for RHEL/CentOS:
sudo yum install -y kubelet-1.32.0-0 kubectl-1.32.0-0 --disableexcludes=kubernetes

# 2. Restart kubelet
sudo systemctl daemon-reload
sudo systemctl restart kubelet

# 3. Verify node version
kubectl get nodes
```

#### For Additional Control Plane Nodes

Repeat the same process on each control plane node, but use:

```bash
sudo kubeadm upgrade node
```

Instead of `kubeadm upgrade apply`.

---

### Step 2: Upgrade Worker Nodes

**Upgrade worker nodes one at a time** to maintain workload availability.

```bash
# On each worker node

# 1. Drain the node (run from control plane)
kubectl drain <worker-node-name> --ignore-daemonsets --delete-emptydir-data

# 2. SSH to the worker node

# 3. Upgrade kubeadm
sudo apt-mark unhold kubeadm && \
sudo apt-get update && \
sudo apt-get install -y kubeadm=1.32.0-00 && \
sudo apt-mark hold kubeadm

# OR for RHEL/CentOS:
sudo yum install -y kubeadm-1.32.0-0 --disableexcludes=kubernetes

# 4. Upgrade the node configuration
sudo kubeadm upgrade node

# 5. Upgrade kubelet and kubectl
sudo apt-mark unhold kubelet kubectl && \
sudo apt-get update && \
sudo apt-get install -y kubelet=1.32.0-00 kubectl=1.32.0-00 && \
sudo apt-mark hold kubelet kubectl

# OR for RHEL/CentOS:
sudo yum install -y kubelet-1.32.0-0 kubectl-1.32.0-0 --disableexcludes=kubernetes

# 6. Restart kubelet
sudo systemctl daemon-reload
sudo systemctl restart kubelet

# 7. Uncordon the node (run from control plane)
kubectl uncordon <worker-node-name>

# 8. Verify node is ready
kubectl get nodes
```

**Repeat for all worker nodes.**

## Post-Upgrade Verification

```bash
# 1. Check cluster version
kubectl version --short

# 2. Verify all nodes are running v1.32
kubectl get nodes -o wide

# 3. Check all pods are running
kubectl get pods -A

# 4. Check system components
kubectl get pods -n kube-system

# 5. Check for any issues
kubectl get events -A --sort-by='.lastTimestamp' | tail -20

# 6. Test a sample workload
kubectl run nginx-test --image=nginx:latest --port=80
kubectl expose pod nginx-test --port=80 --type=NodePort
kubectl get svc nginx-test
# Test access, then cleanup:
kubectl delete pod nginx-test
kubectl delete svc nginx-test

# 7. Verify API resources
kubectl api-resources | grep -E 'gateway|v1'

# 8. Check deprecated API usage (should be zero)
kubectl get --raw /metrics | grep apiserver_requested_deprecated_apis
```

## Rollback Procedure

If the upgrade fails, you can restore from the etcd backup:

```bash
# 1. Stop kube-apiserver
sudo systemctl stop kube-apiserver

# 2. Restore etcd from snapshot
ETCDCTL_API=3 etcdctl snapshot restore /backup/etcd-snapshot-<timestamp>.db \
  --data-dir=/var/lib/etcd-restore

# 3. Update etcd configuration to use restored data
sudo mv /var/lib/etcd /var/lib/etcd.backup
sudo mv /var/lib/etcd-restore /var/lib/etcd

# 4. Restart etcd and kube-apiserver
sudo systemctl restart etcd
sudo systemctl restart kube-apiserver

# 5. Verify cluster state
kubectl get nodes
kubectl get pods -A
```

## Troubleshooting

### Issue: Node Stuck in NotReady State

```bash
# Check node status
kubectl describe node <node-name>

# Check kubelet logs
sudo journalctl -u kubelet -n 100 --no-pager

# Restart kubelet
sudo systemctl restart kubelet
```

### Issue: Pods Not Starting

```bash
# Check pod events
kubectl describe pod <pod-name> -n <namespace>

# Check container runtime
sudo systemctl status containerd
# OR
sudo systemctl status crio

# Check for resource constraints
kubectl top nodes
kubectl top pods -A
```

### Issue: API Server Not Responding

```bash
# Check API server logs
sudo journalctl -u kube-apiserver -n 100 --no-pager

# Check etcd health
ETCDCTL_API=3 etcdctl --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  endpoint health

# Restart API server
sudo systemctl restart kube-apiserver
```

## Best Practices

1. ✅ **Always backup etcd before upgrading**
2. ✅ **Test upgrades in non-production first**
3. ✅ **Upgrade one minor version at a time** (e.g., 1.30 → 1.31 → 1.32)
4. ✅ **Read the changelog before upgrading**
5. ✅ **Maintain version skew policy** (control plane should be no more than 1 minor version ahead of nodes)
6. ✅ **Plan for maintenance windows** with minimal user impact
7. ✅ **Monitor cluster health throughout the upgrade**
8. ✅ **Keep control plane and worker nodes within 1 minor version**

---

## Version Skew Policy

Kubernetes maintains a strict version skew policy:

| Component | Supported Versions |
|-----------|-------------------|
| **kube-apiserver** | N (latest) |
| **kubelet** | N, N-1, N-2 |
| **kube-controller-manager** | N, N-1 |
| **kube-scheduler** | N, N-1 |
| **kube-proxy** | N, N-1, N-2 |
| **kubectl** | N+1, N, N-1 |

**Example**: If API server is v1.32, kubelet can be v1.32, v1.31, or v1.30.

## Summary

✅ Backup etcd before starting
✅ Upgrade control plane first (using `kubeadm upgrade apply`)
✅ Upgrade additional control plane nodes (using `kubeadm upgrade node`)
✅ Upgrade worker nodes one by one
✅ Verify cluster health after each node upgrade
✅ Test workloads thoroughly post-upgrade

**Estimated Downtime**: 0-5 minutes per control plane node (with HA setup), ~1-2 minutes per worker node.
