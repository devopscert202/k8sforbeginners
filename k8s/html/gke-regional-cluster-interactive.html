Google Kubernetes Engine

Interactive Guide to a 3-Node Regional GKE Cluster

GKE is Google Cloud's managed Kubernetes platform. Google runs the control plane for you, integrates deeply with Google Cloud networking and security, and lets you focus on workloads such as WordPress, MySQL, Deployments, Secrets, and Services instead of building the cluster plumbing yourself.

This guide models a regional GKE cluster spread across three zones with one worker node per zone. It shows the managed control plane, kubectl interactions, object deployment flow, and the two main user paths: operators managing the app through kubectl and end users reaching WordPress through a public LoadBalancer Service.

Managed Platform

What GKE gives you on Google Cloud

GKE keeps the Kubernetes API highly available, wires in Google Cloud networking, and automates tasks like upgrades, node provisioning, health checks, and load balancer integration.
Managed Control Plane

Google operates the API server, scheduler, controller manager, and backing control plane infrastructure, so your team works with Kubernetes without manually building the masters.

Regional Resilience

A regional cluster spreads nodes and control plane replicas across zones, reducing the blast radius of a zone outage and improving availability for production workloads.

Cloud-Native Networking

GKE integrates with Google Cloud VPC, health checks, firewall rules, and external load balancers so Services can become reachable with public IPs.

Kubernetes Objects

You still use standard Kubernetes primitives like Pods, Deployments, Secrets, Services, ConfigMaps, and PVCs. GKE manages the platform around them rather than changing the API model.

Regional Topology

3-node regional cluster across three zones

In a regional GKE design, Google keeps the control plane highly available across the region while your worker nodes are distributed into multiple zones for stronger fault tolerance.
Example region: us-central1 with zones us-central1-a, us-central1-b, and us-central1-c.

Regional GKE Control Plane

Google runs multiple control plane replicas for the region and exposes the Kubernetes API endpoint that kubectl talks to. You manage the cluster through the API; you do not SSH into masters or maintain etcd yourself.

If one zone has issues, your other zones still host nodes and reachable application replicas.

Operator Interaction

How kubectl interacts with GKE

kubectl never talks directly to Pods or kubelets first. It talks to the Kubernetes API endpoint, and the control plane makes the rest of the system converge to the declared state.
Step 1

Engineer workstation

User authenticates to Google Cloud, fetches cluster credentials, and uses kubectl locally.

Step 2

GKE API endpoint

kubectl apply -f sends manifests to the managed Kubernetes API server.

Step 3

Controllers reconcile

Deployments create ReplicaSets, Services create endpoints, and Secrets become available.

Step 4

Scheduler places Pods

Pods are assigned to nodes across the three zones based on capacity and policy.

Step 5

Nodes run workloads

Kubelet and container runtime pull images, start containers, and report status back.

1. Connect to the cluster

Get kubeconfig credentials for the regional cluster.

gcloud container clusters get-credentials gke-regional-demo \
  --region us-central1 \
  --project my-gcp-project

2. Create the namespace and secrets

Prepare the namespace where WordPress and MySQL will run.

kubectl create namespace wordpress
kubectl apply -f wordpress-secret.yaml
kubectl get secrets -n wordpress

3. Deploy the workloads

Apply the manifests for MySQL, WordPress, and Services.

kubectl apply -f mysql-deployment.yaml
kubectl apply -f wordpress-deployment.yaml
kubectl apply -f wordpress-service.yaml

4. Verify cluster state

Inspect Pods, Services, and rollout progress.

kubectl get pods -n wordpress -o wide
kubectl get svc -n wordpress
kubectl rollout status deployment/wordpress -n wordpress
kubectl mental model

Think in terms of desired state. When you apply YAML, you are telling Kubernetes what should exist. The control plane and node agents keep working until the running state matches that intent.

Create / update

Use kubectl apply for manifests that describe objects declaratively.

Observe

Use kubectl get, describe, and logs to troubleshoot.

Operate safely

Use rollouts and health checks rather than manually replacing Pods one by one.

Traffic story

Operators manage everything with kubectl, but end users only see the external IP or DNS name of the WordPress load balancer. Inside the cluster, WordPress talks to the MySQL ClusterIP service, not to the pod IP directly.

Secrets usage

Credentials are injected as environment variables rather than hardcoded in images.

Secret

Shared credentials for MySQL and WordPress.

apiVersion: v1
kind: Secret
metadata:
  name: wordpress-db-secret
  namespace: wordpress
type: Opaque
stringData:
  MYSQL_ROOT_PASSWORD: rootpass
  MYSQL_PASSWORD: wordpresspass
  WORDPRESS_DB_PASSWORD: wordpresspass

MySQL Deployment and Service

Internal database endpoint for WordPress.

apiVersion: apps/v1
kind: Deployment
metadata:
  name: mysql
  namespace: wordpress
spec:
  replicas: 1
  selector:
    matchLabels:
      app: mysql
  template:
    metadata:
      labels:
        app: mysql
    spec:
      containers:
      - name: mysql
        image: mysql:8.0
        envFrom:
        - secretRef:
            name: wordpress-db-secret
---
apiVersion: v1
kind: Service
metadata:
  name: mysql
  namespace: wordpress
spec:
  selector:
    app: mysql
  ports:
  - port: 3306
    targetPort: 3306

WordPress Deployment and LoadBalancer Service

Public app entry point backed by multiple replicas.

apiVersion: apps/v1
kind: Deployment
metadata:
  name: wordpress
  namespace: wordpress
spec:
  replicas: 2
  selector:
    matchLabels:
      app: wordpress
  template:
    metadata:
      labels:
        app: wordpress
    spec:
      containers:
      - name: wordpress
        image: wordpress:php8.2-apache
        env:
        - name: WORDPRESS_DB_HOST
          value: mysql
        envFrom:
        - secretRef:
            name: wordpress-db-secret
---
apiVersion: v1
kind: Service
metadata:
  name: wordpress-lb
  namespace: wordpress
spec:
  type: LoadBalancer
  selector:
    app: wordpress
  ports:
  - port: 80
    targetPort: 80
