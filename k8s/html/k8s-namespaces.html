Namespaces

How Kubernetes partitions a single cluster into logical boundaries for teams, environments, and workloads — and what isolation namespaces do and do not provide.

A namespace is a virtual cluster inside a physical cluster. It scopes object names, DNS, RBAC, quotas, and policies — but all namespaces still share the same nodes, network, and control plane. Think of it as a folder system for Kubernetes resources.

Core Model

What Is a Namespace?

Logical isolation

Namespaces group related resources (Pods, Services, Deployments, Secrets) under one name. Two Deployments named web can coexist in different namespaces.

Name scoping

Resource names must be unique within a namespace, not cluster-wide. A Pod named nginx in dev and one in prod are separate objects.

Policy boundary

RBAC Roles, ResourceQuotas, LimitRanges, NetworkPolicies, and Pod Security labels are commonly applied per namespace to govern who can do what and how much.

DNS scoping

Services get fully qualified DNS names: <service>.<namespace>.svc.cluster.local. Short names like myservice resolve only within the same namespace.

Not a security wall

By default, Pods in any namespace can reach Pods in any other namespace over the network. True isolation requires NetworkPolicies, RBAC, and admission controls.

Shared infrastructure

All namespaces share the same worker nodes, control plane, and cluster-scoped resources (Nodes, PersistentVolumes, ClusterRoles). Namespaces do not create separate clusters.

When to use namespaces: Separate teams (team-a, team-b), environments (dev, staging, prod), or application tiers (frontend, backend). Avoid creating hundreds of namespaces — they add management overhead without strong isolation by themselves.
Cluster Layout

How Namespaces Fit in a Cluster

1

Create or select a namespace

Admins create namespaces for teams or environments. Users target a namespace with -n or set a default in kubeconfig.

2

Deploy namespaced resources

Pods, Services, Deployments, ConfigMaps, and Secrets are created inside the namespace. Their names are scoped to that boundary.

3

Apply namespace policies

Attach RBAC Roles, ResourceQuotas, LimitRanges, NetworkPolicies, and Pod Security labels to enforce access and resource rules.

4

Discover via DNS

Pods reach Services in the same namespace by short name. Cross-namespace calls use the full DNS name: service.namespace.svc.cluster.local.

Important: Namespaces provide organizational and policy boundaries, not hard multi-tenancy. For strong tenant isolation, combine namespaces with NetworkPolicies, RBAC, admission controllers, and optionally separate clusters.
YAML and Commands

Working with Namespaces

Create a Namespace
apiVersion: v1
kind: Namespace
metadata:
  name: development
  labels:
    environment: dev
    pod-security.kubernetes.io/enforce: baseline
Deploy to a Namespace
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
  namespace: development    # target namespace
spec:
  replicas: 2
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
      - name: nginx
        image: nginx:1.25
Essential kubectl Commands
# List all namespaces
kubectl get namespaces
kubectl get ns

# Create namespace (imperative)
kubectl create namespace staging

# Run command in a specific namespace
kubectl get pods -n development
kubectl apply -f deployment.yaml -n production

# Set default namespace for current context
kubectl config set-context --current --namespace=development

# View all resources in a namespace
kubectl get all -n development

# Delete a namespace (removes ALL resources inside it)
kubectl delete namespace staging
Namespace with ResourceQuota
apiVersion: v1
kind: ResourceQuota
metadata:
  name: dev-quota
  namespace: development
spec:
  hard:
    pods: "20"
    requests.cpu: "4"
    requests.memory: 8Gi
    limits.cpu: "8"
    limits.memory: 16Gi
Decision Guide

Namespaced vs Cluster-Scoped Resources

Scope Examples Notes
Namespaced Pod, Service, Deployment, ReplicaSet, StatefulSet, DaemonSet, Job, CronJob, ConfigMap, Secret, Ingress, NetworkPolicy, Role, RoleBinding, ResourceQuota, LimitRange Must belong to exactly one namespace. Created with metadata.namespace or -n flag.
Cluster-scoped Node, Namespace, PersistentVolume, StorageClass, ClusterRole, ClusterRoleBinding, CustomResourceDefinition, PriorityClass Exist once per cluster. No namespace field. Visible from all namespaces (subject to RBAC).
Built-in Namespaces

Default System Namespaces

Namespace Purpose Guidance
default Where resources land if no namespace is specified Convenient for learning; avoid in production — always set an explicit namespace
kube-system Core cluster components: CoreDNS, kube-proxy, CNI, metrics-server Do not deploy application workloads here
kube-public Readable by all users; rarely used for workloads Contains cluster info ConfigMap; generally leave empty
kube-node-lease Node heartbeat Lease objects for health detection Managed automatically; do not modify
Quick check: run kubectl api-resources --namespaced=true and kubectl api-resources --namespaced=false to see the full list for your cluster version.
Use It Well

Real-World Patterns and Exercises

Environment separation

Create dev, staging, and production namespaces. Same app name in each — isolated by namespace boundaries and policies.

Team multi-tenancy

Give each team a namespace with its own ResourceQuota and RBAC RoleBinding. Prevents one team from consuming all cluster resources.

Soft vs hard isolation

Namespaces alone = soft isolation (naming + policy). Add NetworkPolicies for network isolation and OPA/Gatekeeper for admission enforcement.

Exercise 1

Create three namespaces: dev, staging, prod. Deploy an nginx Pod named web in each. Verify with kubectl get pods -A.

Exercise 2

Deploy a Service named api in prod. From a Pod in dev, test DNS: short name fails, api.prod resolves.

Exercise 3

Apply a ResourceQuota limiting pods to 5 in dev. Try creating a 6th Pod and observe the admission rejection message.
