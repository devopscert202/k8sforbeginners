🛡️ Kubernetes NetworkPolicy Traffic Filtering

Pod-Level Network Security & Namespace Isolation

📋 Default Behavior (No NetworkPolicy)

⬇️Ingress (Incoming Traffic)

  • Allow-all: All pods accept traffic from any source
  • Cross-namespace traffic: ✅ Allowed
  • No restrictions by default

⬆️Egress (Outgoing Traffic)

  • Allow-all: All pods can send traffic anywhere
  • External connectivity: ✅ Allowed
  • No restrictions by default

🔑 Key Concepts
podSelector: {} Empty selector matches ALL pods in the namespace where the policy is applied
Namespace Isolation Policy blocks ingress from other namespaces while allowing same-namespace traffic
Deny-All Effect Applying a NetworkPolicy with ingress rules creates an implicit deny-all for unmatched traffic
CNI Requirement NetworkPolicy requires CNI plugin support (e.g., Calico, Cilium, Weave Net)
📝 NetworkPolicy Example: Namespace Isolation

deny-from-other-namespaces.yaml

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  namespace: default              # Applied to 'default' namespace
  name: deny-from-other-namespaces
spec:
  podSelector:
    matchLabels: {}                # {} = ALL pods in namespace
  ingress:                          # Define allowed ingress rules
  - from:
      - podSelector: {}            # Allow from ALL pods in same namespace
                                     # (no namespaceSelector = same namespace)
⚙️ How the Policy Works

✅Allowed Traffic

  • Same namespace: pod1 → pod1 (default)
  • Traffic matches podSelector: {}
  • No namespaceSelector = implicit same-namespace

❌Blocked Traffic

  • Cross-namespace: pod2 (test) → pod1 (default)
  • Does not match ingress rules
  • Implicit deny for unmatched traffic
🎯 Advanced Selector Patterns

Combined Pod + Namespace Selector

ingress:
- from:
  - namespaceSelector:
      matchLabels:
        environment: production   # Allow from 'production' namespaces
    podSelector:
      matchLabels:
        role: frontend            # AND pods labeled 'role=frontend'
